# KRAM AI Agent Quality Gate & Engineering Skills

This document defines the strict quality gates, workflow standards, and security constraints for any AI coding agent (e.g., Claude Code, Cursor, Devin, Gemini CLI, Windsurf) operating on the KRAM Auction marketplace repository.

All agents MUST strictly adhere to these guidelines without exceptions.

---

## 🧭 Core Workflow Commands

Agents should guide their software development lifecycle (SDLC) using structured phases:
- **`/spec`** — Create/update specification documents before writing any code.
- **`/plan`** — Formulate a detailed, step-by-step implementation plan including all file diffs and verification tests.
- **`/build`** — Perform compilation and type-safety checks (`npm run build`, `npm run lint`) to ensure zero compile warnings.
- **`/test`** — Execute unit, integration, or E2E tests, verifying success through raw test outputs and CLI logs.
- **`/review`** — Perform security and structural reviews, checking for exposed keys, unhandled database locks, or auth bypasses.
- **`/ship`** — Finalize commits and deployment checks under strict change control.

---

## 🛡️ Rigorous Quality Gates & Security Rules

### 1. Production Secrets & Token Protection
- **No Hardcoded Keys**: Never write actual API keys, database connection strings, JWT secrets, payment tokens, or API credentials inside code files, documentation, walkthroughs, commit messages, or terminal logs.
- **Strict Environment Separation**: Keep production secrets strictly in the Vercel Production Environment dashboard. Use `<CRON_SECRET>` or `<TURBOSMS_TOKEN>` placeholders in all markdown documents.
- **No Console Secret Echoing**: When rotating or setting keys in Vercel environment variables, use blind PowerShell/Terminal pipelines (e.g. dynamically generated GUID variables or direct stdin pipes) so that the actual value is never logged to stdout.

### 2. Database Schema Sync Safety
- **No Dangerous Auto-Pushes**: Do not run `prisma db push` inside compile-time or deployment build scripts (e.g., the Vercel `build` command in `package.json`).
- **Safe Build Scripts**: Keep the production build chain safe and restricted to: `"build": "prisma generate && next build"`.
- **Controlled Migrations**: Database structure synchronization must remain a separate, controlled operational step accompanied by manual/automated database backups.

### 3. Auth Bypass Prevention
- **No Backend Backdoors**: Do not add test-only headers (like `x-test-stress-bypass`) or credential bypass wrappers inside core authentication blocks (such as `auth-config.ts` or `getCurrentUser.ts`) in the production branch.
- **Pure Auth.js Verification**: Session verification must strictly rely on production-grade credentials or official OAuth providers.

### 4. Code & Commit Discipline
- **No Unapproved Pushes**: Do not use `git reset --hard`, `git push origin main`, `git push -f`, `git add -A`, or `git add .` on the production branch without explicit, written confirmation from the repository owner.
- **Evidence-Based Success**: Do not rationalise or assume that an implementation works. Agents must provide verifiable proof of success (e.g., raw JSON responses, compiler logs, and layout screenshots).
- **Direct UI/API Validation**: Do not perform direct database mutations (e.g., Prisma CLI updates) to mock E2E success. Validate only through official public API endpoints and real-user UI interactions.
