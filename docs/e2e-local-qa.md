# Локальне E2E QA Тестування (PostgreSQL)

Цей посібник містить інструкції з налаштування та запуску наскрізного (E2E) тестування в ізольованому середовищі СУБД PostgreSQL. Тестування перевіряє роботу ставок, лімітів, скарг, логування аудиту та модерації користувачів.

---

## ⚠️ ВАЖЛИВІ ПОПЕРЕДЖЕННЯ

> [!WARNING]
> **НІКОЛИ не використовуйте робочий `DATABASE_URL` (production database) для запусків E2E-тестування.** 
> Тести автоматично змінюють дані, міняють терміни аукціонів та створюють фейкові лоти. Використання робочої бази даних призведе до втрати або пошкодження реальних даних користувачів.

> [!IMPORTANT]
> **НЕ коммітьте файл `.env.test.local` у репозиторій Git.**
> Він автоматично доданий до `.gitignore` (через `.env*`), перевіряйте його статус перед кожним комітом.

> [!NOTE]
> **Обмеження швидкості запитів (Rate Limit) та SSE (Realtime) зараз працюють у режимі in-memory MVP.**
> Вони функціонують у рамках оперативної пам'яті одного інстансу розробника та скидаються при перезапуску процесу. Це ідеально для локальної розробки та бета-тесту, але для продакшну потрібна інтеграція із хмарними рішеннями (Redis/Pusher).

## 🛠️ 1. Підготовка тестової бази даних (kram-test-db)

Тестування вимагає ізольованої бази даних PostgreSQL, щоб уникнути пошкодження чи засмічення робочих (production) даних.

### Крок A: Запуск Docker Desktop
1. Переконайтеся, що програма **Docker Desktop** встановлена на вашому комп'ютері.
2. Запустіть **Docker Desktop** з панелі керування Windows або через меню "Пуск".
3. Зачекайте 10–15 секунд, поки статус сервісу в лівому нижньому кутку панелі зміниться на зелений ("Engine running").

### Крок B: Створення контейнера `kram-test-db`
Виконайте наступну команду у терміналі для створення ізольованого тестового інстансу СУБД:

```bash
docker run -d --name kram-test-db \
  -e POSTGRES_USER=kram \
  -e POSTGRES_PASSWORD=<password> \
  -e POSTGRES_DB=kram_test \
  -p 5433:5432 \
  postgres:15
```

*Примітка:* Порт тестової бази навмисно змінено на `5433`, щоб запобігти конфліктам з будь-якими іншими локальними базами даних на порту `5432`.

### Крок C: Конфігурація тестового середовища
Створіть локальний конфігураційний файл `.env.test.local` у корені проекту KRAM зі значеннями:

```ini
DATABASE_URL="postgresql://kram:<password>@localhost:5433/kram_test"
ALLOW_QA_SEED=true
QA_SEED_PASSWORD="<your-local-password>"
NEXTAUTH_SECRET="test-secret"
AUTH_SECRET="test-secret"
NEXTAUTH_URL="http://localhost:3000"
```

## 🚀 2. Синхронізація схеми та сид бази даних

Застосуйте схему даних Prisma та наповніть її початковими тестовими даними (Seed) у вашій новій тестовій базі:

```bash
# 1. Генерація клієнта Prisma
npx prisma generate

# 2. Синхронізація схеми з базою даних (без видалення міграцій)
npx dotenv-cli -e .env.test.local -- npx prisma db push --accept-data-loss

# 3. Наповнення бази тестовими даними (QA seed)
npx dotenv-cli -e .env.test.local -- npm run seed
```

Тестовий сид успішно створить наступні записи у вашому тестовому інстансі PostgreSQL:
1.  **QA Seller:** `qa-seller@kram.local` (роль: продавець)
2.  **QA Buyer:** `qa-buyer@kram.local` (роль: покупець)
3.  **QA Admin:** `qa-admin@kram.local` (роль: адміністратор)
4.  **QA Lot:** лот під назвою `"Тестовий QA-лот KRAM"` із початковою ціною 100 ₴.

## 🏃‍♂️ 3. Запуск dev-сервера та прогін E2E тестування

Для перевірки повного циклу API, валідацій ставок та скарг виконайте:

### Крок A: Запуск dev-сервера
Запустіть локальний сервер розробника, підключений до тестової бази:

```bash
npx dotenv-cli -e .env.test.local -- npm run dev
```

### Крок B: Запуск E2E тестів
У новому вікні терміналу запустіть нативний тестовий сценарій:

```bash
# Для PowerShell (Windows):
$env:DATABASE_URL="postgresql://kram:<password>@localhost:5433/kram_test"
npx tsx scripts/e2e-local-qa.ts

# Для Bash (Linux/macOS):
DATABASE_URL="postgresql://kram:<password>@localhost:5433/kram_test" npx tsx scripts/e2e-local-qa.ts
```

## 📊 4. Інтерпретація результатів PASS/FAIL

Кожен крок тесту відображає свій статус. Усі **12 тестів** мають повертати **PASS**:

### Що вважається PASS:
*   **QA Seller/Buyer/Admin exists (Тести 1–3):** Успішно знайдені користувачі з відповідними email у базі.
*   **QA Lot exists (Тест 4):** Успішно знайдено лот `"Тестовий QA-лот KRAM"`.
*   **Seller cannot bid on own lot (Тест 5):** Спроба зробити ставку на свій лот відхилена помилкою з кодом `CANNOT_BID_OWN` або повідомленням про власний лот.
*   **Bid below minimum rejected (Тест 6):** Спроба зробити ставку нижче мінімального ліміту (наприклад, менше поточної ціни + крок) успішно заблокована.
*   **Valid bid creates Bid (Тест 7):** Ставка створює запис у таблиці `Bid`, оновлюється поточна ціна лота (`Listing.currentPrice`) та лічильник ставок (`Listing.bidCount`).
*   **Ended auction blocks bids (Тест 8):** Bids на лот із завершеним часом успішно блокуються на рівні сервера.
*   **Report creation (Тест 9):** Створено скаргу із текстовим коментарем у таблиці `Report` у статусі `PENDING`.
*   **Admin report REVIEWED (Тест 10):** Статус скарги успішно змінено на `REVIEWED` адмін-командою.
*   **AuditLog created (Тест 11):** Лог із типом `REPORT_REVIEWED` та метаданими успішно створено в `AuditLog` базі.
*   **Admin user verification (Тест 12):** Статус верифікації продавця змінюється з `false` на `true`, плашка довіри реактивно оновлюється.

### Що вважається FAIL:
*   Якщо хоча б один тест повертає `❌ FAIL` (наприклад, ставка від імені продавця пройшла успішно, або ставка нижче кроку не заблокувалася).
*   Процес завершується з ненульовим exit-кодом (`1`), що зупиняє CI/CD pipeline у разі інтеграції.
